# 1️⃣ Floating Point Formats

## 📚 How Computers Store Decimal Numbers (IEEE 754)

Computers can't store "1.5" directly. Instead, they use a formula called **IEEE 754**:

$$\text{value} = (-1)^{\text{sign}} \times 2^{(\text{exponent} - \text{bias})} \times (1 + \text{mantissa})$$

Think of it like scientific notation: $6.022 \times 10^{23}$

| Part | What It Does | Analogy |
|------|-------------|----------|
| **Sign** (1 bit) | Positive or negative | The ± button on a calculator |
| **Exponent** | How big/small the number is (the power) | The "×10²³" part — sets the scale |
| **Mantissa** | The actual digits of precision | The "6.022" part — the detail |

> 🎨 **Visual: How bits are divided in each format:**
>
> ```
> FP32:  [1 sign] [8 exponent] [23 mantissa]  = 32 bits total
> FP16:  [1 sign] [5 exponent] [10 mantissa]  = 16 bits total
> BF16:  [1 sign] [8 exponent] [ 7 mantissa]  = 16 bits total
> FP8:   [1 sign] [4 exponent] [ 3 mantissa]  =  8 bits total (E4M3)
> FP8:   [1 sign] [5 exponent] [ 2 mantissa]  =  8 bits total (E5M2)
> ```

⭐ **Key Pattern**: More exponent bits = wider range (bigger/smaller numbers). More mantissa bits = more decimal precision.

## 📊 Format Comparison Table

| Format | Bits | Exponent | Mantissa | Range | Precision |
|--------|------|----------|----------|-------|-----------|
| FP32 | 32 | 8 | 23 | ±3.4×10³⁸ | ~7 digits |
| FP16 | 16 | 5 | 10 | ±65,504 | ~3.3 digits |
| BF16 | 16 | 8 | 7 | ±3.4×10³⁸ | ~2.4 digits |
| FP8 (E4M3) | 8 | 4 | 3 | ±448 | ~1 digit |
| FP8 (E5M2) | 8 | 5 | 2 | ±57,344 | ~0.6 digits |

## 🔍 Deep Dive Into Each Format

### 📏 FP32 (Full Precision) — "The Gold Standard"
- **32 bits** = 4 bytes per number
- Range: $\pm 3.4 \times 10^{38}$ — can represent astronomically large AND tiny numbers
- Precision: ~7 decimal digits (e.g., $3.1415927$)
- 🏭 **Production**: Used as the "master copy" of weights during training. All optimizer states (Adam's momentum, variance) live in FP32
- ⚠️ **Downside**: Slow on modern GPU Tensor Cores, uses lots of memory

### 📐 FP16 (Half Precision) — "The Pocket Ruler"
- **16 bits** = 2 bytes per number (half of FP32!)
- Range: only $\pm 65{,}504$ — much smaller range!
- Precision: ~3.3 decimal digits (e.g., $3.14$)
- 🏭 **Production**: Used on older GPUs (V100 era). Requires **loss scaling** because tiny gradients ($10^{-8}$) underflow to zero
- ⚠️ **Downside**: Limited range causes overflow/underflow headaches

### 📐 BF16 (Brain Float 16) — "The Wide-Range Ruler" ⭐
- **16 bits** = 2 bytes — same size as FP16
- Range: $\pm 3.4 \times 10^{38}$ — **same range as FP32!** (because it has 8 exponent bits)
- Precision: ~2.4 decimal digits (e.g., $3.1$) — less precise than FP16
- 🏭 **Production**: **The modern default!** Google designed BF16 specifically for TPU/AI training. No loss scaling needed!
- ⭐ **Why BF16 wins**: The range matters more than precision for training. Gradients can be very large OR very tiny — BF16 handles both.

> 📖 **Citation**: Google introduced BF16 for Cloud TPU v2 (2018). The "B" stands for "Brain" — from the Google Brain team.

### 🔍 FP8 (Quarter Precision) — "The Thumbnail Ruler"
- **8 bits** = 1 byte — the smallest practical format
- Two variants:
  - **E4M3**: Range ±448, ~1 digit precision — better for weights/activations
  - **E5M2**: Range ±57,344, ~0.6 digits — better for gradients
- 🏭 **Production**: Emerging for inference on NVIDIA H100 (Hopper architecture). Experimental for training.

> 📖 **Citation**: NVIDIA et al. (2022) — *"FP8 Formats for Deep Learning"* whitepaper — established E4M3/E5M2 as the standard FP8 variants.

## 🎯 Dynamic Range Comparison (LaTeX)

The **dynamic range** of a format is determined by its exponent bits:

$$\text{Dynamic Range} = \left[2^{-(2^{e-1}-2)}, \; 2^{2^{e-1}-1}\right]$$

where $e$ = number of exponent bits.

| Format | Exponent Bits ($e$) | Smallest Positive Normal | Largest Value |
|--------|------|---|---|
| FP32 | 8 | $\approx 1.18 \times 10^{-38}$ | $\approx 3.4 \times 10^{38}$ |
| FP16 | 5 | $\approx 6.1 \times 10^{-5}$ | $65{,}504$ |
| BF16 | 8 | $\approx 1.18 \times 10^{-38}$ | $\approx 3.4 \times 10^{38}$ |

⭐ **Notice**: BF16 and FP32 have identical dynamic range because they share the same 8 exponent bits!

**FP16**: More precision, smaller range → needs loss scaling
**BF16**: Same range as FP32, less precision → simpler, no loss scaling needed
**BF16 is preferred** for modern training (H100, A100).

# 2️⃣ Why Mixed Precision?

## 🏠 The Analogy: Rough Work vs. Precise Records

> 🎨 **Think of it like building a house:**
> - The **construction workers** do rough measurements with a tape measure (FP16/BF16) — fast and good enough for cutting wood
> - The **architect's blueprints** keep precise measurements to 7 decimal places (FP32) — needed for structural integrity
> - **Mixed precision** = do the rough work fast (FP16), but keep precise records (FP32) for the important stuff

## 📊 Benefits Table

| Benefit | Mechanism |
|---------|-----------|
| 2x memory savings | Weights/activations stored in 16-bit |
| 2x faster compute | Tensor Cores operate on FP16/BF16 |
| Larger batches | Halved memory → double batch size |
| Same quality | Critical ops stay in FP32 |

"Mixed" = FP16 for most ops + FP32 for sensitive ops (reductions, norms, softmax).

### 🏭 Production Impact (Real Numbers)

| Metric | FP32 Only | Mixed Precision | Improvement |
|--------|-----------|----------------|-------------|
| Training time (GPT-3 175B) | ~60 days | ~34 days | **1.8× faster** |
| GPU memory per model copy | 700 GB | 350 GB | **50% less** |
| Max batch size (80GB GPU) | 2 | 4-8 | **2-4× larger** |
| Model quality (perplexity) | Baseline | ~Same | **No degradation** |

> ⭐ **Every modern training run uses mixed precision.** If you're training in pure FP32, you're wasting half your compute budget!

> 📖 **Citation**: Micikevicius et al. (2018) — *"Mixed Precision Training"* demonstrated that mixed FP16/FP32 training matches FP32 quality across CNNs, RNNs, and GANs while providing 2-3× speedup.

# 3️⃣ The Loss Scaling Problem (FP16 Only)

## 🔍 The Magnifying Glass Analogy

> 🎨 **Imagine you're trying to read tiny text on a page:**
> - The text is so small (like tiny gradients, $\approx 10^{-8}$) that your eyes can't see it
> - FP16 is like your eyes — it can't "see" numbers smaller than $\approx 6 \times 10^{-5}$
> - **Loss scaling** = using a magnifying glass 🔍 to make the tiny text BIGGER so you can read it
> - After reading, you mentally "un-magnify" to get the real size

## ⚠️ The Problem: Underflow

FP16 range: 2⁻²⁴ to 65,504.
Gradients can be tiny (e.g., 10⁻⁸) → underflow to zero in FP16!

$$\text{Tiny gradient: } g = 3.2 \times 10^{-8} \xrightarrow{\text{FP16 cast}} 0 \quad \text{(LOST! 😱)}$$

When gradients become zero, the model **stops learning** — this is catastrophic!

## ✅ Solution: Dynamic Loss Scaling

The math behind loss scaling:

$$\text{Step 1: } \mathcal{L}_{\text{scaled}} = \mathcal{L} \times S \quad \text{(scale up the loss, e.g., } S = 2^{16} = 65536\text{)}$$

$$\text{Step 2: } \nabla_{\text{scaled}} = \nabla(\mathcal{L}_{\text{scaled}}) = S \cdot \nabla(\mathcal{L}) \quad \text{(gradients are scaled up too, by chain rule)}$$

$$\text{Step 3: } \nabla_{\text{real}} = \frac{\nabla_{\text{scaled}}}{S} \quad \text{(divide back to get true gradients)}$$

**Solution: Dynamic Loss Scaling**
1. Multiply loss by a large scale factor (e.g., 2¹⁶ = 65,536)
2. All gradients are scaled up proportionally
3. Before optimizer step, divide gradients by the scale factor
4. If overflow detected (inf/nan), skip the step and reduce scale

### 🔄 Visual Walkthrough

```
  Tiny gradient: 3.2×10⁻⁸
       │
       ▼
  ❌ FP16 cast → 0 (underflow! gradient lost!)

  WITH Loss Scaling (S = 65536):

  Loss × 65536 → Scaled gradient: 3.2×10⁻⁸ × 65536 = 0.002097
       │
       ▼
  ✅ FP16 cast → 0.002097 (fits! gradient preserved!)
       │
       ▼
  Divide by 65536 → 3.2×10⁻⁸ (original gradient recovered ✅)
```

> ⭐ **Dynamic** means the scale factor $S$ adjusts automatically: it starts large and shrinks if it causes overflow (inf/nan). PyTorch's `GradScaler` handles this for you.

> 💡 **Why BF16 doesn't need this**: BF16 has 8 exponent bits (same as FP32), so it can represent numbers as small as $\approx 10^{-38}$. Tiny gradients don't underflow!

# 4️⃣ PyTorch AMP (Automatic Mixed Precision)

## 🤔 What Is AMP?

> 🎨 **Analogy**: AMP is like having a smart assistant who automatically decides:
> - "This calculation is fine with rough numbers" → uses FP16/BF16 (fast! 🏎️)
> - "This calculation needs precision" → uses FP32 (accurate! 🎯)
>
> You don't have to manually convert every operation — `autocast` does it for you!

## 📝 FP16 Training Code (With Loss Scaling)

```python
from torch.cuda.amp import autocast, GradScaler

# Training with AMP
model = MyModel().cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)
scaler = GradScaler()  # Dynamic loss scaling (for FP16)

for epoch in range(num_epochs):
    for x, y in train_loader:
        x, y = x.cuda(), y.cuda()
        optimizer.zero_grad()
        
        # Forward pass in mixed precision
        with autocast(dtype=torch.float16):
            output = model(x)
            loss = criterion(output, y)
        
        # Backward with scaled loss
        scaler.scale(loss).backward()
        
        # Unscale gradients, clip, then step
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        scaler.step(optimizer)
        scaler.update()  # Adjust scale factor
```

### 🔍 Line-by-Line Explanation for Beginners

| Line | What It Does | Why |
|------|-------------|-----|
| `from torch.cuda.amp import autocast, GradScaler` | Imports two key tools | `autocast` auto-picks precision; `GradScaler` handles loss scaling |
| `scaler = GradScaler()` | Creates the "magnifying glass" 🔍 | Manages dynamic scale factor $S$ to prevent underflow |
| `with autocast(dtype=torch.float16):` | Enters mixed-precision zone | Everything inside runs in FP16 (fast!) except sensitive ops (auto FP32) |
| `scaler.scale(loss).backward()` | Scales loss UP then backprops | Multiplies loss by $S$ so tiny gradients survive FP16 casting |
| `scaler.unscale_(optimizer)` | Divides gradients by $S$ | Recovers true gradient values before the optimizer uses them |
| `scaler.step(optimizer)` | Updates weights (skips if inf/nan) | If overflow detected, this step is skipped entirely |
| `scaler.update()` | Adjusts $S$ for next iteration | Increases $S$ if no overflow; decreases if overflow detected |

> ⭐ **The whole flow**: Scale up → backward → scale down → check for overflow → update weights

### ⚠️ Common Pitfalls

| Mistake | Symptom | Fix |
|---------|---------|-----|
| Forgetting `scaler.unscale_()` before grad clipping | Clipping wrong values | Always unscale before `clip_grad_norm_` |
| Using FP16 without GradScaler | Loss goes to NaN | Always pair FP16 with GradScaler |
| FP16 for accumulation | Loss of precision | Use `dtype=torch.float32` for reduction ops |

# 5️⃣ BF16 Training (Simpler)

BF16 has the same exponent range as FP32 → no loss scaling needed!

> 🎨 **Why BF16 is simpler**: Remember the magnifying glass analogy? BF16's "eyes" can already see tiny numbers (down to $10^{-38}$), so you don't need the magnifying glass (GradScaler) at all!

```python
# BF16 training (preferred on A100/H100)
for x, y in train_loader:
    x, y = x.cuda(), y.cuda()
    optimizer.zero_grad()
    
    with autocast(dtype=torch.bfloat16):
        output = model(x)
        loss = criterion(output, y)
    
    loss.backward()  # No scaler needed!
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    optimizer.step()
```

### 🆚 FP16 vs BF16 Training Side-by-Side

| Aspect | FP16 Training | BF16 Training |
|--------|--------------|---------------|
| Needs GradScaler? | ✅ Yes (required!) | ❌ No |
| Lines of code | ~10 extra lines | Standard loop |
| Risk of NaN | Higher (overflow) | Very low |
| GPU support | V100, A100, H100 | A100, H100, TPU |
| Industry preference | Legacy | ⭐ **Modern default** |

> 🏭 **Production Note**: LLaMA, GPT-4, PaLM, Gemini — virtually all modern LLMs train in BF16. It's simpler, more stable, and just as fast.

# 6️⃣ What Stays in FP32?

## 🤔 Why Can't Everything Be FP16/BF16?

> 🎨 **Analogy**: Imagine a kitchen:
> - **Chopping vegetables** (matrix multiplication) — a rough chop is fine, speed matters → FP16 ✅
> - **Measuring medicine doses** (normalization, softmax) — you NEED exact amounts → FP32 ✅
>
> Some operations are "numerically sensitive" — small rounding errors compound into big mistakes.

autocast automatically handles this, but understanding helps debugging:

| Op Category | Precision | Why |
|------------|-----------|-----|
| MatMul, Conv | FP16/BF16 | Tensor Cores acceleration |
| BatchNorm, LayerNorm | FP32 | Numerically sensitive |
| Softmax | FP32 | Large reductions overflow |
| Loss functions | FP32 | Accumulation accuracy |
| Optimizer state | FP32 | Momentum, variance need precision |

### 📐 Why Softmax Needs FP32 (Math Example)

Softmax computes:

$$\text{softmax}(x_i) = \frac{e^{x_i}}{\sum_j e^{x_j}}$$

If $x_i = 100$, then $e^{100} \approx 2.69 \times 10^{43}$ — this **overflows FP16** (max 65,504) but fits fine in FP32!

### 🔄 The Master Weights Pattern

The **master weights** (a FP32 copy) are maintained for the optimizer update:
```
FP16 forward → FP16 gradients → FP32 weight update → copy to FP16
```

> 🎨 **Analogy**: You draft notes quickly in pencil (FP16) but maintain the official ledger in ink (FP32). The pencil draft is fast, but the ink copy is the source of truth.

# 7️⃣ Memory Breakdown Example

## 🧮 Understanding the Math

> 💡 **Quick rule**: 1 parameter in FP32 = 4 bytes. In FP16 = 2 bytes.
> So a 7B model: $7 \times 10^9 \times 4 \text{ bytes} = 28 \text{ GB}$ in FP32.

For a 7B parameter model:

| Component | FP32 | Mixed Precision |
|-----------|------|----------------|
| Model weights | 28 GB | 14 GB (FP16) + 28 GB (FP32 master) |
| Gradients | 28 GB | 14 GB |
| Optimizer (Adam) | 56 GB | 56 GB (FP32 states) |
| Activations | Variable | Halved |
| **Total** | ~112 GB | ~84 GB (+ much faster) |

### 🤔 Wait, Mixed Precision Uses MORE Memory for Weights?

Yes! You keep BOTH copies:
- FP16 weights for fast forward/backward passes
- FP32 master weights for precise optimizer updates

But you **save huge** on gradients + activations, and get **2× speed** from Tensor Cores.

> 🏭 **Production Reality**: For LLM training, the memory savings on activations is the biggest win. A single transformer layer's activations can be tens of GB — halving that is massive.

With techniques like ZeRO (next days), this is distributed across GPUs.

# 8️⃣ Practical Tips

## 🛠️ Essential Checks Before Training

```python
# 1. Check if your GPU supports BF16
print(torch.cuda.is_bf16_supported())  # True for Ampere+

# 2. Profile memory savings
def profile_training_step(model, x, use_amp=False):
    torch.cuda.reset_peak_memory_stats()
    
    if use_amp:
        with autocast(dtype=torch.bfloat16):
            loss = model(x).sum()
        loss.backward()
    else:
        loss = model(x).sum()
        loss.backward()
    
    peak = torch.cuda.max_memory_allocated() / 1e9
    print(f"AMP={use_amp}: Peak memory = {peak:.2f} GB")

# 3. Verify no quality degradation
# Compare FP32 vs BF16 validation loss over training
```

### 🏭 Production Decision Guide

| Scenario | Recommendation | Why |
|----------|---------------|-----|
| Training on A100/H100 | BF16 (no scaler) | Simplest, most stable |
| Training on V100 | FP16 + GradScaler | V100 doesn't support BF16 |
| Fine-tuning a pretrained model | BF16 | Fewer numerical surprises |
| Inference deployment | FP16 or FP8 | Maximize throughput |
| Debugging NaN losses | Temporarily switch to FP32 | Isolate precision issues |
| Training with very deep networks | BF16 > FP16 | Deep nets have wider gradient range |

### ⚠️ Overflow vs. Underflow Quick Reference

| Problem | What Happens | When It Occurs | Solution |
|---------|-------------|----------------|----------|
| **Overflow** 💥 | Number too large → becomes `inf` | FP16 value > 65,504 | Reduce loss scale; use BF16 |
| **Underflow** 🕳️ | Number too small → becomes `0` | FP16 gradient < $6 \times 10^{-5}$ | Loss scaling; use BF16 |
| **NaN** ❌ | `inf - inf` or `0/0` | Overflow in intermediate ops | Check loss scale; reduce LR |
